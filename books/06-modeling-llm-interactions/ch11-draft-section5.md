## The API Is the Mental Model

A developer who uses lm15 for six months develops a way of thinking about LLM interactions. They think in `complete()` calls and `model()` objects. They think in `resp.text` and `resp.tool_calls`. They think in `tools=[function]` and `system="instruction"`. They think in `stream.text` and `reasoning=True`. These concepts become the vocabulary of their internal monologue about LLMs — the words they use when they think about what they're building, when they explain it to colleagues, when they debug it at midnight.

This vocabulary is not neutral. It shapes what the developer considers natural, what they consider possible, and what they never think to try.

A developer who thinks in `complete()` — a single function that takes a prompt and returns text — naturally thinks of LLM interactions as request-response. One question, one answer. This is the right model for classification, extraction, and summarization. It's the wrong model for an agent that acts over multiple turns, but the developer whose vocabulary starts with `complete()` must *discover* the agent model — it's not in their default vocabulary.

A developer who thinks in `model()` — a stateful object with memory — naturally thinks of LLM interactions as conversations. Multi-turn, accumulating context, building understanding. This is the right model for chatbots, research sessions, and interactive tools. It's the wrong model for batch processing 10,000 documents, but the developer whose vocabulary starts with `model()` must discover the batch model — their default is conversational.

A developer who thinks in LangChain's `Chain` and `Agent` — orchestrated sequences of calls with planning and reflection — naturally thinks of LLM interactions as workflows. Directed graphs of operations, each feeding into the next. This is the right model for complex pipelines. It's the wrong model for a simple API call, but the developer whose vocabulary includes chains and agents sees simple calls as degenerate cases of workflows, not as the fundamental operation.

None of these vocabularies is wrong. Each one makes certain thoughts easy and certain thoughts hard. `complete()` makes the simple call effortless and the agent pattern invisible. `Chain` makes the workflow explicit and the simple call ceremonial. **The API doesn't reflect the developer's mental model. The API creates it.** A developer using lm15 thinks differently about LLM interactions than a developer using LangChain, not because they have different knowledge but because they have different vocabulary.

This is the Sapir-Whorf hypothesis applied to programming: the language you work in shapes the thoughts you can think. A strong version — that the API *determines* thought — is too strong. Developers can think beyond their tools. But a weak version — that the API *influences* thought, making some ideas natural and others effortful — is clearly true. The developer who has `tools=[function]` in their vocabulary reaches for function-calling sooner than the developer who must construct `Tool` objects with JSON Schema. The convenience doesn't just reduce keystrokes. It reduces the cognitive cost of the idea, which means the idea occurs more often and earlier in the design process.

The implication for API design is profound and uncomfortable: **you're not just building a library. You're building the way your users think about the domain.** The names you choose become their concepts. The abstractions you provide become their mental models. The things you hide become the things they don't consider. The things you surface become the things they reach for first.

`resp.text` surfaces text. The developer reaches for text. Tool calls, thinking traces, citations — these are secondary attributes, accessed through less prominent properties. The API taught the developer that text is primary. A different API — one where the response is a list of parts, and you must choose which parts to read — would teach a different lesson: that all content types are equal, and text is just one of them. The lesson might be more accurate. It would also be less convenient. The tension between accuracy and convenience is permanent, and the resolution is a statement about what you want the developer to think.

This is the deepest layer of API design — deeper than naming, deeper than progressive disclosure, deeper than pits and cliffs. It's the question of what reality you're constructing for the developer. An LLM interaction is, objectively, a complex thing: a multimodal, multi-turn, multi-step, probabilistic, provider-specific interaction with a remote system. The API reduces this complexity to something the developer can hold in their head. The reduction is necessary — nobody can think about all the complexity at once. The question is what you keep and what you drop, and that question is a question about values.

lm15 values simplicity. The simple call is one line. The response is `.text`. The provider switch is a string change. The tool is a function. The complexity exists — ten chapters of this book examined it — but it's beneath the surface, accessible when needed, invisible when not. The developer's mental model is simple because the API presents simplicity. Whether the developer *should* have a simple mental model — whether simplicity serves them or misleads them — is a question the API can't answer. It can only make its choice and live with the consequences.

Every library makes this choice. LangChain chooses expressiveness — chains, agents, memory systems, output parsers — and the developer's mental model is rich and complex. LiteLLM chooses compatibility — OpenAI's format, everywhere — and the developer's mental model is OpenAI's. Vercel chooses integration — React hooks, streaming UI primitives — and the developer's mental model is web-native. lm15 chooses simplicity — one function, one response, one string to switch models — and the developer's mental model is small and transferable.

The choice is the library's identity. Not the dependency count (Chapter 7), not the layer architecture (Chapter 8), not the concealment strategy (Chapter 9). The mental model the API creates is what the library *is* to the developer who uses it. Everything else is implementation.

This book has examined ten design problems in the modeling of LLM interactions — content, conversation, tools, normalization, adapters, streaming, dependencies, boundaries, concealment, time — and each one was ultimately in service of this: the mental model the developer carries. The types are shaped to be thinkable. The boundaries are drawn to be invisible. The convenience properties exist to match the developer's first question. The escape hatches exist because the developer's understanding will eventually exceed the model the API provides.

The API is not the surface of the library. It's the language the developer thinks in. Design it with the knowledge that you're not just building software. You're building a way of seeing.
