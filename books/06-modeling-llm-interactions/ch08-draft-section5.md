## The Instability Insight

Traditional layering advice — from Parnas to Clean Architecture to hexagonal architecture — assumes a common change pattern: business logic changes fast, infrastructure changes slow. The database API is stable. HTTP is stable. The file system is stable. You draw boundaries around the fast-changing parts (domain logic, application rules) and depend on the slow-changing parts (database drivers, HTTP clients, OS interfaces). The boundaries protect the volatile from the stable.

LLM systems invert this.

The "infrastructure" — provider APIs — changes as fast as anything else in the system. OpenAI has evolved their API through three major versions: completions, chat completions, and the responses API. Anthropic has changed their message format, added tool calling, added extended thinking, and changed their cache control mechanism — each change affecting the wire format that adapters must produce and parse. Gemini has changed endpoint paths between API versions and restructured their content format. New providers emerge every few months (Mistral, Cohere, DeepSeek, Grok), each with their own wire format that requires a new adapter.

In a traditional system, the adapter layer would be the most stable. You write the database adapter once and forget about it. In an LLM system, the adapter layer is one of the most actively changing surfaces. The adapters are updated more frequently than the conversation model, more frequently than the streaming implementation, more frequently than the middleware. The "infrastructure" is volatile.

This changes the layering calculus in a specific way. The adapter boundary — between the client (stable) and the adapters (volatile) — becomes the most critical boundary in the system. Not because it's the most architecturally interesting (it isn't — it's a straightforward Protocol contract). Because it absorbs the most change. Every provider API update, every new provider, every wire format evolution hits the adapter side of this boundary and is contained there. The client, the Model, the sugar, the types — none of them feel it.

If the boundary were elsewhere — if provider-specific logic were mixed into the client or the Model — every provider change would ripple through the system. Adding thinking support for Anthropic would require changing the Model's tool execution loop. Adding cache control for Gemini would require changing the client's dispatch logic. The volatility would be uncontained, and the system would be in permanent churn.

The boundary's job isn't to be elegant. It's to be the *right place* for change to occur. In traditional systems, the right place for change is the business logic layer, because business logic changes fastest. In LLM systems, the right place for change is the adapter layer, because provider APIs change fastest. lm15's five-layer architecture works not because five is the architecturally correct number of layers, but because it places the fastest-changing surface behind the narrowest boundary.

This is the LLM-specific insight about layering: **draw boundaries based on what changes, not on what is.** The structural decomposition — "this is HTTP, this is translation, this is orchestration" — is the starting point. The temporal decomposition — "this changes monthly, this changes quarterly, this changes yearly" — determines which structural boundaries earn their keep. A boundary between two things that change at the same rate and for the same reasons is overhead. A boundary between two things that change at different rates is load-bearing. In LLM systems, the adapter boundary is load-bearing because provider APIs and library logic change at radically different rates and for completely different reasons.
