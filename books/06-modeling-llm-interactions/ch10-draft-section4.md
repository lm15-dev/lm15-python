## Contract Stability

The adapter Protocol — `complete(LMRequest) → LMResponse` and `stream(LMRequest) → Iterator[StreamEvent]` — is the stability promise. Third-party adapters depend on it. Plugin authors code to it. Breaking it breaks the ecosystem.

But what does "breaking" mean? The contract has three layers, and each layer has different stability properties.

**The methods** are the most stable. `complete()` takes a request and returns a response. `stream()` takes a request and yields events. These signatures have never changed in lm15's history and are unlikely to change — they capture the essential interaction pattern (send a request, get a response, possibly incrementally) that all providers share. A plugin author who depends only on the method signatures is safe.

**The types** are mostly stable. `LMRequest` has gained fields — `config.reasoning` was added when reasoning support arrived. `Usage` gained `cache_read_tokens` and `reasoning_tokens`. `Part` gained new type values (citation, refusal). Each addition is backward-compatible: new fields have defaults, new type values are additive. A plugin that constructs an `LMRequest` with the fields that existed when it was written still works — the new fields default to `None` or empty. A plugin that inspects a response still works — the new fields are additional, not replacement.

But backward-compatible doesn't mean invisible. A plugin that parses `resp.message.parts` and switches on `part.type` will encounter types it doesn't recognize — `"citation"`, `"refusal"` — if it was written before those types existed. The plugin should handle unknown types gracefully (the discriminator pattern's `case _: pass`), but a plugin that uses `isinstance` checks against a fixed set of Part subclasses (the hierarchy pattern) would silently drop unknown types. The choice between discriminator and hierarchy (Chapter 1, Section 3) has contract-stability implications that extend beyond the library itself into the plugin ecosystem.

**The semantics** are the least stable. What does `finish_reason="stop"` mean? On the surface: the model finished generating. But the semantics are subtler — does it mean "the model chose to stop" or "the model hit a natural ending" or "the generation reached a provider-defined stopping point"? These distinctions don't matter most of the time. They matter when you're debugging an agent that stops unexpectedly, or when you're comparing behavior across providers, or when a provider changes what "stop" means in a new API version.

`prompt_caching=True` is a semantic contract: "reduce cost on repeated prefixes." The mechanism behind it has changed (Anthropic updated their cache control format) and will change again. A plugin that depends on the mechanism — inspecting request bodies for `cache_control` markers — breaks when the mechanism changes. A plugin that depends on the semantic — setting `prompt_caching=True` and checking `usage.cache_read_tokens` — survives.

The principle for plugin authors: **depend on the most stable layer of the contract.** Methods first (most stable), then core type fields (mostly stable), then semantics of universal parameters (less stable), then escape hatch keys (unstable). The deeper your dependency, the more frequently you'll need to update. A plugin that depends on method signatures updates never. A plugin that depends on escape hatch keys updates every time the provider changes something.

For the library author, the principle is different: **evolve the contract by addition, not by modification.** New fields with defaults. New type values. New optional methods. Never change the meaning of an existing field. Never remove a type value. Never change a method signature. The contract grows monotonically — it gets larger but never changes what's already there. This is the frozen-dataclass principle applied to time: the type is immutable within a version, and evolution adds to it rather than modifying it.
