# lm15 benchmarks

All numbers below are generated by `benchmarks/suite/run.py` and read
from `benchmarks/RESULTS.json` — nothing in this file is hand-typed.

## Methodology

- Machine: Intel(R) Core(TM) i7-1065G7 CPU @ 1.30GHz (8 CPUs), Linux 6.8.0-124-generic
- Python: 3.13.3 · uv 0.9.26 · run at 2026-06-11T01:33:43+00:00
- Each competitor is installed into its own scratch venv under
  `/tmp/lm15-bench-venvs/` (`uv venv` + `uv pip install`); lm15 is
  installed from the locally built wheel. Failures are recorded and the
  suite continues.
- **Cold import** = wall time of `python -c "import X"` in a fresh
  process, best of 10 (after one warmup run to
  populate the page cache and `.pyc` files). A bare `python -c pass`
  baseline is shown for scale.
- **Install size** = bytes added to `site-packages` vs an empty venv,
  measured after byte-compilation (i.e. what sits on disk after first use).
  **Transitive deps** = installed package count minus the package itself.
- **Import RSS** = median `VmRSS` after the import minus a bare
  interpreter in the same venv.
- **Hot path** (lm15 only) = median ns/op over 2000
  timed iterations after 200 warmup
  iterations, in-process. Response bodies and the SSE transcript are
  real pinned captures from `lm15-contract/bodies/`.
- **TTFR tax** = time-to-first-byte of a streaming generate request to
  local ollama, lm15's `StdlibTransport` vs raw `urllib.request`,
  identical request bytes, interleaved runs, median of N.

## TTFR tax vs raw urllib (local ollama)

Model `qwen3.5:0.8b`, median of 9 interleaved runs:

| client | TTFB | total |
|---|---:|---:|
| lm15 StdlibTransport | 623.9 ms | 1417.8 ms |
| raw urllib.request | 708.4 ms | 1708.4 ms |

lm15 transport tax over raw stdlib HTTP: **-84.5 ms** TTFB.
Run-to-run TTFB spread: lm15 573 ms, urllib 656 ms — the model server's
own jitter dwarfs any difference between the two clients; the tax
is indistinguishable from zero.

## Why lm15 is small and fast

- **Zero runtime dependencies.** The whole library is stdlib-only
  (`websockets` is one optional extra for live sessions). There is no
  pydantic, no httpx/requests, no certifi chain to install, resolve, or
  import.
- **No pydantic.** Canonical types are plain `dataclass`es with exact,
  hand-written serde — no metaclass machinery or validators executing at
  import time or per call.
- **Stdlib transport.** HTTP goes through `http.client`/`urllib`-level
  plumbing the interpreter already loaded, so the wire layer adds
  essentially nothing over raw stdlib HTTP.
